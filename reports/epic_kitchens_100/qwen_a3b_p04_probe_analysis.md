## 结论

这轮输出可以作为“候选事件表/候选记忆”的第一阶段产物，但还不能直接作为 benchmark 真值。模型能抓住厨房、主要器具、烹饪流程、部分状态变化；问题主要在四类：单视频事实被泛化成“通常/习惯”、时间段较粗、局部动作遗漏、少量物体和状态变化误判。

`P04_29` 没有得到最终 JSON，不是视频输入链路失败，而是 `finish_reason=length`：2048 个输出 token 都被推理字段消耗，`message.content` 为 `null`。需要重跑并提高 `max_tokens`，或在服务/模型侧关闭推理输出。

## 分视频判断

### P04_106

模型抓住了主线：揉面、覆盖面团、开烤箱、处理烤盘/烤制食物。官方标注也显示该段包含大量 `dough`、`tea towel`、`oven dish`、`foil`、`pork belly` 相关动作。

主要问题：

- 模型把后半段概括成“用铝箔纸覆盖食物并放回烤箱”，但官方标注中更细的是取出烤盘、拿开锡纸、用勺子处理/品尝猪五花肉、清理台面，最后再把烤盘放回烤箱。
- 模型遗漏了“品尝/吃猪五花肉块”“厨房纸擦手/擦炉灶”等对记忆问答有价值的动作。
- `memory_candidates` 中“烤盘通常放置在烤箱下层”“使用铝箔纸覆盖烤制食物”有泛化风险。单段视频只能写成“本段视频中烤盘从烤箱中取出，之后放回烤箱”。

### P04_24

模型抓住了大体烹饪流程：称量花生、研磨香料、制作椰浆/调料、切牛肉。官方标注支持这些事件：`measure peanuts`、`grind spices`、`scrape spices into pan`、`mix coconut powder`、`cut beef`。

主要问题：

- 开头时间段有误：模型说 `00:00-00:23` 打开冰箱取食材，但官方标注开头是开水龙头、洗手、拿锅、放慢炖锅碗、打开炉灶。冰箱相关动作出现在后面。
- 模型遗漏了后段“把花生放入烤箱”“设置计时器”“搅拌牛肉”等事件。
- “坚果袋存放在冰箱中”可能来自后续 `put away peanuts`，但问题提示写成“通常存放在哪里”不严谨。应改成“本段视频中花生被放回冰箱”。

### P04_29

这段需要重跑。现有推理字段显示模型大致识别到了：看冰箱、搅拌锅、从香料抽屉取调料、加椰子粉、看菜谱、开柜子、处理糖/容器、清理刀和锅、从冷冻柜取鱼饼/米饭等。

与官方标注对比，推理中有一些偏差：

- 官方标注有 `put pan in slow cooker`、`move rice cooker`、`plug slow cooker in`、`put lid on slow cooker`，模型推理里弱化了慢炖锅和电饭锅状态变化。
- 推理里提到“肉包放进冰箱”，官方标注更接近“牛肉包装放入垃圾桶、冲洗牛肉包装、香茅和南姜放入冰箱”。
- 官方标注后段有“取金枪鱼饼放入锅、找米饭、把米饭放入微波炉、启动微波炉”，这类跨设备状态变化适合进入记忆候选。

## 对 benchmark 构建的启发

建议把 VLM 输出定位成候选层，而不是最终标注层：

1. 单段视频先抽取事件表、状态变化和候选记忆。
2. 用官方 EPIC 动作标注做自动对齐，过滤明显不在视频中的动作和物体。
3. 对跨会话记忆，必须经过二次聚合：同一厨房布局、同一物体位置、同一流程在多个 session 重复出现后，才能升级为“习惯/通常/长期事实”。
4. 评测问题应保留证据时间戳，并记录来自哪个 session，避免模型只凭常识回答。

## 下次重跑建议

`P04_29` 至少把输出 token 提到 4096。当前本地 bundle 的 `run_probe.sh` 已经支持：

```bash
BASE_URL=http://127.0.0.1:8000/v1 \
MODEL=qwen35-a3b \
VIDEO_ID=P04_29 \
FPS=1 \
MAX_TOKENS=4096 \
./run_probe.sh
```

如果仍然只输出推理、不输出 JSON，再尝试 `MAX_TOKENS=8192`，或在模型侧关闭推理输出。
