# 分层视频证据抽取与质检流水线设计

## 目标

构建一条可复现、可审计、可恢复的 EPIC-KITCHENS-100 隐藏参考证据生产流水线。流水线使用集群中的本地视觉语言模型完成高召回率分层抽取，使用确定性规则和百炼 API 完成高精确率质检，最终只把经过放行的事实用于参考证据图、评测问题和标准答案生成。

这条流水线服务于 benchmark 数据构建，不向被评测智能体暴露隐藏证据，也不规定被评测方法如何构建记忆。

## 核心原则

1. 原始模型输出永不覆盖，清洗、复核、修正和放行结果写入独立目录。
2. 每条事实必须能回溯到参与者、原始视频、时间范围和上游抽取记录。
3. 抽取侧优先保证召回率，质检侧优先保证精确率；证据不足时拒绝放行，不强行补全。
4. 聚合层不得把未出现于下层的事实变成新的观察证据。
5. 单个会话只能产生“待跨会话验证”的候选，不能直接形成长期习惯或偏好结论。
6. 模型置信度不直接作为放行依据；放行由结构校验、独立视觉复核和人工抽检共同决定。
7. 所有输出记录模型、提示词、参数和流水线版本，支持论文复现和后续版本比较。

## 总体数据流

```text
EPIC-KITCHENS-100 完整 540p/16fps 代理视频（COS）
  -> 集群按 30 秒临时切分
  -> 本地 35B 视觉模型抽取 micro evidence
  -> micro 确定性校验
  -> 120 秒 window 文本聚合
  -> window 确定性校验
  -> 完整 source video/session 文本聚合
  -> session 确定性校验和候选规范化
  -> 百炼按原始视频分组进行候选视觉复核
  -> 争议候选生成 10-30 秒临时证据片段并二次复核
  -> 人工分层抽检与争议裁决
  -> reference-ready 证据库
  -> 跨会话证据图、问题和标准答案构建
```

## 阶段一：源视频与清单

### 输入

- 每个参与者的完整代理视频，默认规格为 540p、16fps、保留音频。
- 代理视频的 COS 对象键和可刷新签名地址。
- 每个视频的 `participant_id`、`source_video_id`、时长和校验信息。

### 约束

- COS 中只长期保存完整代理视频，不长期保存 30 秒、120 秒或质检片段。
- 签名地址属于运行时产物，不写入 Git。
- 清单中的 `source_video_id` 是所有层级的稳定外键。

## 阶段二：30 秒局部事实抽取

集群在推理前按 30 秒切分完整代理视频，尾段短于 10 秒时与前一段合并。本地 Qwen3.5-35B-A3B 以 1fps 读取片段，输出物体、原子事件、状态观察、状态变化、结束状态和不确定项。

每条局部事实必须包含：

- `clip_id`；
- 相对当前 30 秒片段的时间范围；
- 相关实体或事件标识；
- 事实文本；
- 置信度；
- 不确定性说明。

模型原始响应和清洗后的 JSON 分开保存。JSON 解析失败或输出截断时允许幂等重试，不覆盖已经存在的成功结果。

## 阶段三：确定性结构校验

三层输出在进入下一层前都经过本地规则校验。规则不判断画面语义真假，只判断结构、来源和逻辑约束。

### 通用检查

- 必填字段完整且类型正确；
- 标识在当前记录内唯一；
- 时间范围合法且没有超出父级范围；
- 枚举值属于允许集合；
- 子实体、事件、窗口和片段引用存在；
- 数量不超过提示词约束；
- 输出能回溯到源视频和父级记录；
- 不确定项引用的候选存在。

### 会话级附加检查

- 包含“习惯、通常、总是、长期、偏好、喜欢”等长期结论措辞时标记 `long_term_overclaim`；
- 候选受到不确定项影响时标记 `affected_by_uncertainty`，不得保持高置信；
- 只有一个窗口支持的稳定布局或流程候选标记 `single_window_support`；
- 聚合层出现下层没有来源的事实标记 `unsupported_aggregation`；
- 非典型物体位置、过细的容器类型或不可读屏幕内容标记 `needs_visual_review`。

校验结果写入 `quality_flags`，不修改上游原始文件。结构错误进入重试队列；语义风险进入百炼复核队列。

## 阶段四：120 秒窗口聚合

每四个左右连续的 30 秒记录组成一个 120 秒窗口，不跨越原始视频边界。窗口聚合只接收结构校验通过的 micro evidence；被标记为不确定的事实可以保留，但必须携带不确定状态。

输出包括局部实体映射、事件链、状态变化、窗口结束状态、证据事实和冲突。所有聚合事实必须保存 `supporting_clip_ids`。

## 阶段五：完整会话聚合

当前 EPIC-KITCHENS 处理阶段把一条完整原始视频视为一次连续会话。会话层聚合所有窗口，形成会话时间线、实体、状态更新、结束状态、未完成任务和跨会话候选。

`cross_session_evidence_candidates` 只是待验证候选。每条候选必须包含：

- `candidate_id`；
- 观察事实和观察值；
- `supporting_window_ids`；
- 可解析出的支持时间范围；
- 后续跨会话需要验证的内容；
- 置信度和质量标记。

## 阶段六：百炼整段视频分组复核

### 请求粒度

第一轮质检按 `source_video_id` 分组。一条百炼请求输入：

- 已上传 COS 的完整 540p/16fps 代理视频地址；
- 当前会话中全部待复核候选，最多 12 条；
- 每条候选的支持窗口和绝对时间范围；
- 固定的三分类判断协议。

因此 P30 的 25 条源视频对应约 25 个批量请求，而不是为 206 条候选重复输入完整视频。

### 模型与接口

- 默认模型：`qwen3.7-plus`；
- 接口：百炼兼容 OpenAI 的 Batch API；
- 思考模式：关闭；
- 温度：0；
- API Key：只从 `DASHSCOPE_API_KEY` 环境变量读取；
- 输入 JSONL 不包含密钥；
- `custom_id` 使用稳定的 `source_video_id`，用于合并结果。

### 判定协议

每个候选必须返回：

```json
{
  "candidate_id": "memcand_1",
  "verdict": "entailed|contradicted|insufficient",
  "corrected_claim": null,
  "evidence_time_ranges": [{"start_sec": 0, "end_sec": 10}],
  "reason_codes": ["location_not_confirmed"],
  "confidence": "high|medium|low"
}
```

- `entailed`：画面足以支持原候选；
- `contradicted`：画面明确反驳原候选；
- `insufficient`：抽帧、遮挡、时间定位或画面清晰度不足，无法裁决。

复核模型不得依据常识补全，不得因为原抽取标为高置信而提高自己的判断，也不得把修正描述直接作为已放行事实。

## 阶段七：局部证据二次复核

以下候选进入二次复核：

- 第一轮返回 `insufficient`；
- 第一轮返回 `contradicted`；
- 第一轮输出缺失、格式错误或候选未覆盖；
- 完整视频超过接口限制；
- 规则标记为异常位置、细粒度状态变化或其他高风险事实；
- 人工抽检发现需要重看局部画面。

流水线根据支持窗口和模型给出的时间范围，从完整代理视频生成 10-30 秒临时片段。临时片段上传到独立 COS 前缀，生成覆盖批量任务执行期的签名地址，完成二次复核后按清理清单删除。

二次复核仍使用 `entailed|contradicted|insufficient`。只有二次复核明确支持时，原候选才可以进入自动放行；模型给出的 `corrected_claim` 必须进入人工审核，不能自动替换原候选。

## 阶段八：候选状态机与放行规则

候选状态只允许沿以下路径变化：

```text
extracted
  -> schema_failed
  -> schema_passed
      -> verification_passed
      -> verification_rejected
      -> verification_uncertain
          -> local_verification_passed
          -> local_verification_rejected
          -> human_review_required
              -> human_accepted
              -> human_rejected
```

自动进入参考证据库必须同时满足：

1. 三层结构和引用校验通过；
2. 没有 `long_term_overclaim` 或 `unsupported_aggregation`；
3. 百炼复核结果为 `entailed`；
4. 复核证据时间范围与候选支持范围有交集；
5. 不受尚未解决的冲突或不确定项影响。

最终记录增加：

```json
{
  "qc_status": "verification_passed",
  "usable_for_reference": true,
  "quality_flags": [],
  "extractor_model": "qwen35-a3b",
  "verifier_model": "qwen3.7-plus",
  "pipeline_version": "v0.2"
}
```

## 阶段九：人工抽检

P30 试跑阶段对全部会话级候选进行人工核验，用于估计百炼复核的精确率和错误类型。全量运行阶段采用分层抽检：

- 所有模型分歧、证据不足和修正候选全部检查；
- 所有异常位置、长期措辞和高风险状态变化全部检查；
- 从自动放行候选中按参与者、事实类型、视频时长和置信度分层抽样；
- 任何将直接用于 benchmark 问题和标准答案的事实，在题目发布前再次检查。

参考证据生产以精确率优先。抽检发现系统性错误时，暂停对应参与者或事实类型的放行，修订提示词或规则后重新处理受影响记录。

## 输出目录与可恢复性

建议目录结构：

```text
outputs/epic_kitchens_100/p30_micro_30s/
outputs/epic_kitchens_100/p30_windows_120s/
outputs/epic_kitchens_100/p30_sessions_full/
outputs/epic_kitchens_100/p30_qc/
  validation/
  bailian_batch_inputs/
  bailian_batch_outputs/
  merged/
  local_review_queue/
  human_review/
  reports/
```

每个阶段使用稳定标识幂等运行。成功结果默认跳过，失败项可按标识重试。所有批量提交记录保存任务 ID、模型、参数、输入文件哈希、提示词哈希、提交时间和结果文件路径。

## 质量报告

每次运行至少输出以下统计：

- 三层记录数、解析成功率和结构校验通过率；
- 候选总数及各事实类型分布；
- 规则风险标记数量和比例；
- 百炼 `entailed`、`contradicted`、`insufficient` 比例；
- 二次复核触发率和最终裁决分布；
- 自动放行率、人工复核率和拒绝率；
- 抽取模型与复核模型一致率；
- 人工抽检精确率及按事实类型分层的错误率；
- 从 micro 到 session 的错误传播案例数量。

## P30 验收标准

在扩展到完整数据集前，P30 必须满足：

1. 25 条会话和全部候选都能生成稳定的百炼批量请求；
2. 百炼结果能按 `candidate_id` 无损合并回原候选；
3. 任一候选都能回溯到原始视频和支持时间范围；
4. 批量任务中断、部分失败和重复运行不会覆盖成功结果；
5. 已知的“刀具放入冰箱”错误被规则或视觉复核拦截，不能进入参考证据库；
6. 人工抽检报告能够量化放行候选的精确率；
7. API 密钥、签名地址和视频文件不进入 Git。

## 暂不纳入本版本

- 自动生成 benchmark 问题和标准答案；
- 跨参与者实体对齐；
- 用百炼替代本地全量 micro 抽取；
- 自动接受复核模型生成的修正事实；
- 对音频语义进行正式质检。当前视觉复核不验证语音内容，音频事实后续通过语音识别或全模态模型单独处理。
